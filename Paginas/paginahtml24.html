Lectores de pantalla.

Son softwares que mapean las páginas que usamos y las lee en voz alta.

Existe una combinación de lectoresde pantalla con navegadores que hacen más eficaz la experiencia a los usuarios:

Lectores Navegadores
NVDA Mozilla Firefox
JAWS Microsoft Internet Explorer
VoiceOver Safari
ChromeVox Google Chrome

Accessibility tree.

En el inspector de elementos en la sección de Elementos se encuentra una subsección en donde se muestran los estilos, en esta subsección se puede seleccionar la ficha Accessibility y se muestra un árbol de accesibilidad o accessibility tree.

El acessibility tree es un mapeo que se crea mientras se carga la página cogiendo todo el HTML semantico y muestra la estructura de la página y los elementos dentro de ella, en el accessibility tree se conectan los lectores de pantallas para leer el sitio web y saber que tienen que leer y en que orden. Al seleccionar un elemento del árbol se puede visualizar los contenidos del elemento, su rol y si es enfocable por medio del teclado o por lectores de pantallas.

Con esto se puede verificar que la estructura de la página web facilite la lectura a los lectores de pantalla y el teclado.

ARIA

A veces el HTML semántico no es suficiente para las aplicaciones de accesibilidad sobre todo paara páginas interactivas que utilizan JavaScript, para estos casos se utiliza ARIA.

ARIA ( Accessible Rich Internet Applications ): Un conjunto de atributos especiales para accesibilidad que pueden añadirse a cualquier etiqueta, pero especialmente adaptado a HTML.

Fue creado por la W3C, lo crearon para que podamos comunicar cambios especiales en nuestras aplicaciones.

Atributos de ARIA:

  • Roles
  • Propiedades
  • Estados

Roles de ARIA

Define el tipo general del objeto (como un artículo, una alerta o un deslizador).

Los roles le comunican al navegadores cuales son las interacciones que debería esperar, cómo se va a usar este objeto en el proyecto y cómo se navega entre las secciones del proyecto. Es mejor depender del HTML semántico para comunicar los roles, por lo que los roles se deben utilizar en situaciones muy especificas.

Propiedades de ARIA

Comunican atributos que son esenciales para el comportamiento o significado de un elemento pero que suelen comunicar visualmente como las flechas que despliegan un carrusel de imágenes o un switch que cambia el tema de la página.

Con el atributo aria-label="" se puede agregar una descripción a los elementos que solamente se comunican de forma visual, de esta forma los lectores de pantalla pueden leer en voz alta estos elementos.

Estados de ARIA

Comunican estados y cambios de estados en elementos que se suelen comunicar visualmente como esconder elementos del lector que también estan escondidos visualmente para los usuarios.

Con el atributo aria-hidden="true" se pueden esconder elementos al lector de pantalla, ejemplo con un carrusel de imágenes: ul > li:aria-hidden > button > img

Cuando se aplica ARIA a interacciones, se suele utilizar con JavaScript, el método element.setAttribute("aria-hidden", "true"); para colocar el atributo a una etiqueta o con el método element.removeAttribute("aria-hidden"); para remover el atributo de la etiqueta.

Con el atributo aria-live=" off | polite | assertive " se puede establecer la prioridad con la que el lector de pantalla debe tratar a los cambios dinámicos que surgen.

  • off es la configuración predeterminada.
  • polite hace que el lector de pantalla termine de leer el elemento actual para leer el cambio dinámico. Se utiliza cuando el mensaje que se quiere comunicar, no es importante.
  • assertive interrumpe la lectura del elemento actual para leer el cambio dinámico. Se utiliza cuando se quiere dar prioridad al mensaje.

Algunos navegadores necesitan que el elemento con aria-live se coloque como el elemento padre del elemento dinámico, ejemplo:

<div aria-live="assertive" >
 <div> ¡El mensage acaba de ser mandado correctamente! </div>
</div>
